# 📔 개발 레슨 런 (Lessons Learned & Retrospective)

이 문서는 프로젝트 개발 과정에서 발생한 시행착오, 설계상의 아쉬운 점, 그리고 이를 통해 배운 교훈들을 기록하는 공간입니다. 같은 실수를 반복하지 않고 더 견고한 시스템을 만들기 위한 가이드라인으로 활용합니다.

---

### 🕒 [2026-03-13] 알림 오발송 및 학부모 수신 혼란 이슈

#### 1. 발생한 문제 및 한계 (Errors & Limitations)
- **데이터 무결성 미흡**: `parent_token` 생성 마이그레이션 시 `UNIQUE` 제약 조건을 명시적으로 걸지 않아, 아주 낮은 확률로 토큰이 중복될 수 있는 설계적 허점이 있었음.
- **알림 전략의 불일치**: 백엔드 서버(교사 페이지)는 모든 이석 승인 시 알림을 보내지만, 프론트엔드(학부모 페이지)는 외출/외박만 보여주는 '정보의 비대칭'이 발생하여 사용자의 오해를 불러일으킴.
- **구독 관리의 맹점**: 한 기기(브라우저)에서 여러 학생의 링크를 클릭할 경우, 이전 구독 정보를 정리하지 않고 계속 누적하여 '한 기기 다수 수신' 현상이 발생함.

#### 2. 설계상의 교훈 (Design Insights)
- **"보이지 않는 것은 알리지 마라"**: 백엔드 알림 로직은 항상 프론트엔드의 데이터 노출 범위와 동기화되어야 함. (불필요한 과잉 알림 지양)
- **보수적인 데이터 관리**: 고유 식별자(Token, ID 등)를 생성할 때는 코드 레벨의 랜덤 생성에만 의존하지 말고, 반드시 DB 레벨에서 고유성을 보장할 것.
- **사용자 실수 방어**: 사용자가 실수로 다른 링크를 클릭하는 등의 예외 상황을 고려하여, 기기당 수신 정보의 '정정(Sync)' 기능을 고려해야 함.

#### 3. 향후 적용할 규칙 (Action Items)
- ✅ 알림 발송 시 `leave_type` 필터링 적용 (외출/외박 등 주요 항목 선별).
- ✅ 기기 구독 시 불필요한 중복이 쌓이지 않도록 구독 갱신 로직 검토.
- ✅ 데이터베이스 스키마 변경 시 제약 조건(Unique, Not Null 등) 재검토.
